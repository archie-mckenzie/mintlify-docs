ワークスペースのルールとメモリを活用して、Windsurf をスタイルガイドやコンポーネント、プロジェクトのコンテキストを理解するドキュメントのエキスパートに育てましょう。

<div id="using-windsurf-with-mintlify">
  ## Mintlify と Windsurf の併用
</div>

Windsurf の Cascade AI アシスタントは、Mintlify コンポーネントを使って、組織の基準に沿ったドキュメントを書くように調整できます。Workspace ルールと Memories はプロジェクトに関する持続的なコンテキストを提供し、Cascade の提案の一貫性を高めます。

* **Workspace ルール** はドキュメントリポジトリに保存され、チームで共有されます。
* **Memories** は時間とともに蓄積される個人単位のコンテキストを提供します。

共有のドキュメント標準に合わせるため、Workspace ルールの設定を推奨します。作業しながら Memories を育てることはできますが、共有されないため、チームメンバー間で一貫しません。

ドキュメントリポジトリの `.windsurf/rules` ディレクトリで Workspace ルールを作成します。詳細は Windsurf ドキュメントの [Memories & Rules](https://docs.windsurf.com/windsurf/cascade/memories) を参照してください。

<div id="example-workspace-rule">
  ## ワークスペースルールの例
</div>

このルールは、Mintlifyのコンポーネントと一般的なテクニカルライティングのベストプラクティスに関する前提情報をCascadeに提供します。

このサンプルルールはそのまま使うことも、ドキュメントに合わせてカスタマイズすることもできます。

* **執筆基準**: 言語ガイドラインを自社のスタイルガイドに合わせて更新する。
* **コンポーネントパターン**: プロジェクト固有のコンポーネントを追加する、または既存の例を調整する。
* **コード例**: 汎用例は、プロダクトの実際のAPI呼び出しとレスポンスに置き換える。
* **スタイルとトーンの指針**: 用語、書式、その他のルールを調整する。

作成したルールは、ドキュメントリポジトリの`.windsurf/rules`ディレクトリに`.md`ファイルとして保存します。

````mdx
# Mintlify technical writing rule

## Project context

- これはMintlifyプラットフォーム上のドキュメントプロジェクトです
- MDXファイルとYAMLフロントマターを使用します  
- ナビゲーションは `docs.json` で設定します
- テクニカルライティングのベストプラクティスに従います

## Writing standards

- 手順は二人称（"you"）で記述する
- 能動態・現在時制で記述する
- 手順は前提条件から始める
- 主要なステップには期待される結果を含める
- 説明的でキーワードを含む見出しを使う
- 文は簡潔かつ有益に保つ

## Required page structure

すべてのページはフロントマターから始めます:

```yaml
---
title: "明確で具体的なタイトル"
description: "SEOとナビゲーションのための簡潔な説明"
---
```

## Mintlify components

### docs.json

- サイトナビゲーションとdocs.jsonの作成時は [docs.json schema](https://mintlify.com/docs.json) を参照する

### Callouts

- 補足的で有用な情報には `<Note>`
- 重要な注意点や破壊的変更には `<Warning>`
- ベストプラクティスや専門的アドバイスには `<Tip>`  
- 中立的なコンテキスト情報には `<Info>`
- 成功の確認には `<Check>`

### Code examples

- 適切な場合は、完全に実行可能な例を含める
- 複数言語の例には `<CodeGroup>` を使用する
- すべてのコードブロックに言語タグを指定する
- ダミーではなく現実的なデータを使用する
- APIドキュメントでは `<RequestExample>` と `<ResponseExample>` を使用する

### Procedures

- 逐次手順には `<Steps>` コンポーネントを使用する
- 必要に応じて `<Check>` コンポーネントで検証ステップを含める
- 複雑な手順は小さなステップに分割する

### Content organization

- プラットフォーム別の内容には `<Tabs>` を使用する
- 段階的開示には `<Accordion>` を使用する
- 重要な内容の強調には `<Card>` と `<CardGroup>` を使用する
- 画像は説明的な代替テキスト付きで `<Frame>` コンポーネントに包む

## API documentation requirements

- すべてのパラメータは `<ParamField>` で記述する
- レスポンス構造は `<ResponseField>` で示す
- 成功とエラーの両方の例を含める
- 入れ子のオブジェクトのプロパティには `<Expandable>` を使用する
- 常に認証の例を含める

## Quality standards

- 公開前にすべてのコード例をテストする
- 内部リンクには相対パスを使用する
- すべての画像に代替テキストを付与する
- 見出し階層を適切に保つ（h2から開始）
- 既存のパターンとの一貫性を確認する
````

<div id="working-with-cascade">
  ## Cascade の活用
</div>

ルールを設定したら、Cascade を使って各種ドキュメント作業を効率化できます。詳細は Windsurf ドキュメントの [Cascade](https://docs.windsurf.com/windsurf/cascade) を参照してください。

<div id="example-prompts">
  ### 例のプロンプト
</div>

**新規コンテンツの作成**:

```text wrap
自社APIの認証方法を説明する新しいページを作成してください。JavaScript、Python、cURL のコード例を含めてください。
```

**既存コンテンツの改善**:

```text wrap
このページを見直し、わかりやすさとコンポーネントの活用に関する改善点を提案してください。手順をより追いやすくすることに注力してください。
```

**コード例の作成**:

```text wrap
このAPIエンドポイントのエラーハンドリングを示す完結したコード例を生成してください。現実的なデータを使用し、想定されるレスポンスも含めてください。
```

**一貫性の維持**:

```text wrap
この新しいページが社内のドキュメント標準に準拠しているか確認し、必要な修正があれば提案してください。
```
