Git は、ドキュメントの変更を追跡し、チームメンバーと共同作業できるバージョン管理システムです。Git を使えば、プロジェクト内の各ファイルについて、何が、いつ、なぜ変更されたのかを確認できます。必要に応じて、ファイルを以前のバージョンに戻すことも簡単です。

ウェブエディターは、バックグラウンドでいくつかの Git 操作を実行します。これらの概念を理解しておくと、ウェブエディターをより効果的に使え、ローカル開発環境で作業するチームメンバーとの協業も円滑になります。

<div id="core-git-concepts">
  ## Git の基本概念
</div>

<AccordionGroup>
  <Accordion title="Repository">
    ドキュメントのソースで、すべてのファイルとその履歴が保存されます。ウェブエディターはリポジトリに接続してコンテンツへアクセス・変更します。
  </Accordion>

  <Accordion title="Commit">
    ある時点の変更を保存したスナップショットです。各コミットには変更内容を説明するメッセージが含まれ、プロジェクト履歴に恒久的な記録を残します。
  </Accordion>

  <Accordion title="Branch">
    ブランチは、リポジトリ内の特定のコミットを指します。公開中のドキュメントはデプロイブランチからビルドされます。未公開の変更を含むブランチをいくつでも作成できます。ブランチの変更を公開中のドキュメントに取り込みたい場合は、プルリクエストでそのブランチをデプロイブランチにマージします。

    ブランチを使うと、公開中のドキュメントに影響を与えずに作業でき、新機能を安全に試し、公開前にレビューを受けられます。
  </Accordion>

  <Accordion title="Deployment branch">
    公開中のドキュメントがビルドされるプロジェクトの主要ブランチです。このブランチへの変更は自動的にドキュメントサイトへ公開されます。一般的には `main` と呼ばれますが、任意のブランチをデプロイブランチとして設定できます。
  </Accordion>

  <Accordion title="Pull request">
    ブランチの変更を公開中のドキュメントにマージするための提案方法です。変更が公開される前にレビューと議論を行えます。一般的に PR と呼ばれ、GitLab では merge request とも呼ばれます。
  </Accordion>

  <Accordion title="Diff">
    diff（差分）は、ファイルの2つのバージョン間の変更を示します。プルリクエストをレビューする際、diff は追加・削除・修正点を強調表示し、何が変わったかを把握しやすくします。
  </Accordion>
</AccordionGroup>

<div id="how-the-web-editor-uses-git">
  ## ウェブエディターがGitをどのように使うか
</div>

ウェブエディターは [GitHub App](/ja/settings/github) または [GitLab integration](/ja/settings/gitlab) を通じてGitリポジトリに接続し、一般的なGit操作を自動化します。

次の操作を行うと:

* **ファイルを開く**: エディターがリポジトリから最新の版を取得し、常に最新のコンテンツで作業できます。
* **変更を加える**: エディターが変更をドラフトとして追跡し、保存の準備ができたらコミットにできます。
* **変更を保存する**: エディターが変更を含むコミットを作成し、プロジェクトの履歴に作業を残します。
* **ブランチを作成する**: エディターがリポジトリに新しいブランチを作成します。リポジトリにアクセスできるメンバーなら誰でもそのブランチで共同作業やレビューができます。
* **デプロイブランチに公開する**: エディターがデプロイブランチに直接コミットしてプッシュし、変更を即時に公開します。
* **他のブランチに公開する**: エディターがプルリクエストを作成し、デプロイブランチにマージする前に他の人からフィードバックを得られるようにします。

<div id="git-best-practices">
  ## Git のベストプラクティス
</div>

各チームには独自のワークフローや好みがありますが、まずは次の一般的なベストプラクティスから始めましょう。

* **わかりやすいコミットメッセージを書く**: 何をどう変えたかを、能動的で具体的に記述します。
* **意味の伝わるブランチ名を使う**: ブランチで行う作業内容が伝わり、リポジトリのブランチ一覧を見た人にも意味がわかる名前にします。
* **ブランチの目的を絞る**: 変更は特定のタスクやプロジェクトに集中させます。
* **マージ後はブランチを削除する**: 不要になったブランチは削除し、リポジトリを整理された状態に保ちます。
