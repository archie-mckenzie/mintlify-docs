<div id="introduction">
  ## Einführung
</div>

Der Web-Editor ist eine visuelle Oberfläche zum Erstellen, Bearbeiten und Überprüfen von Dokumentation direkt im Browser.

- **Visuelles Bearbeiten**: Nimm Änderungen an deiner Dokumentation mit einem WYSIWYG-Editor (What You See Is What You Get) vor, der zeigt, wie deine Inhalte bei der Veröffentlichung aussehen.
- **Git-Synchronisierung**: Alle Änderungen werden automatisch mit deinem Git-Repository synchronisiert, um die Versionskontrolle sicherzustellen.
- **Zusammenarbeit in Echtzeit**: Mehrere Teammitglieder können gleichzeitig an der Dokumentation arbeiten; Änderungen sind für alle sichtbar.
- **Keine Einrichtung erforderlich**: Starte direkt vom Dashboard aus mit dem Schreiben.

<div id="web-editor-flow">
  ### Ablauf im Web-Editor
</div>

So arbeitest du typischerweise im Web-Editor:

<Steps>
  <Step title="Wähle deinen Branch">
    Erstelle einen Branch oder nimm Änderungen direkt an deinem Deployment-Branch vor.
  </Step>

  <Step title="Öffne deine Datei">
    Navigiere zu einer bestehenden Datei oder erstelle eine neue.
  </Step>

  <Step title="Bearbeite deinen Inhalt">
    Nimm Änderungen im Web-Editor entweder im visuellen Modus oder im Markdown-Modus vor.
  </Step>

  <Step title="Vorschau deiner Änderungen">
    Sieh dir an, wie deine Änderungen im visuellen Modus erscheinen.
  </Step>

  <Step title="Veröffentliche deine Änderungen">
    Wenn du an deinem Deployment-Branch arbeitest, veröffentliche deine Änderungen direkt im Web-Editor. Auf anderen Branches veröffentlichst du deine Änderungen über einen Pull Request.
  </Step>
</Steps>

<div id="visual-mode">
  ### Visueller Modus
</div>

Der visuelle Modus bietet ein WYSIWYG-Erlebnis, bei dem die Änderungen, die Sie im Editor vornehmen, genau so veröffentlicht werden, wie sie auf Ihrer Dokumentationsseite erscheinen. Dieser Modus ist ideal, wenn Sie in Echtzeit sehen möchten, wie Ihre Änderungen aussehen.

<div id="markdown-mode">
  ### Markdown-Modus
</div>

Der Markdown-Modus bietet direkten Zugriff auf den zugrunde liegenden `MDX`-Code Ihrer Dokumentation. `MDX` kombiniert Markdown-Syntax mit React-Komponenten und gibt Ihnen die volle Kontrolle über die Struktur Ihrer Inhalte. Dieser Modus ist ideal, wenn Sie präzise Kontrolle über Komponenten­eigenschaften benötigen oder wenn Sie es vorziehen, in Markdown-Syntax zu schreiben.

<div id="git-fundamentals">
  ## Git-Grundlagen
</div>

Der Web-Editor führt im Hintergrund Git-Operationen aus. Das Verständnis dieser Konzepte hilft Ihnen, effektiver mit dem Web-Editor zu arbeiten und mit Teammitgliedern zusammenzuarbeiten, die in ihren lokalen Umgebungen arbeiten.

<AccordionGroup>
  <Accordion title="Repository">
    Die Quelle Ihrer Dokumentation, in der alle Dateien und ihre Historie gespeichert sind. Der Web-Editor verbindet sich mit Ihrem Repository, um auf Inhalte zuzugreifen und diese zu bearbeiten.
  </Accordion>

  <Accordion title="Commit">
    Ein gespeichertes Abbild Ihrer Änderungen zu einem bestimmten Zeitpunkt.
  </Accordion>

  <Accordion title="Branch">
    Ein separater Arbeitsbereich, um Änderungen vorzunehmen, ohne Ihre Live-Dokumentation zu beeinträchtigen. Denken Sie daran als sichere Sandbox für Experimente und größere Aktualisierungen.
  </Accordion>

  <Accordion title="Deployment branch">
    Der Hauptbranch, der die Inhalte Ihrer Live-Dokumentation enthält. Änderungen an diesem Branch werden automatisch auf Ihrer Dokumentationsseite veröffentlicht. Häufig als `main` bezeichnet.
  </Accordion>

  <Accordion title="Pull request">
    Eine Möglichkeit, das Zusammenführen Ihrer Branch-Änderungen in Ihre Live-Dokumentation vorzuschlagen. Ermöglicht Überprüfung und Diskussion, bevor Änderungen live gehen. Häufig als PR bezeichnet.
  </Accordion>

  <Accordion title="Diff">
    Ein Diff (oder Unterschied) zeigt die konkreten Änderungen zwischen zwei Versionen einer Datei. Beim Prüfen von Pull Requests heben Diffs hervor, was hinzugefügt, entfernt oder geändert wurde.
  </Accordion>
</AccordionGroup>

<div id="what-the-web-editor-automates">
  ### Was der Web-Editor automatisiert
</div>

Der Web-Editor verbindet sich über unsere [GitHub App](/de/settings/github) oder [GitLab-Integration](/de/settings/gitlab) mit Ihrem Git-Repository und übernimmt Git automatisch. Wenn Sie:

- **Eine Datei öffnen**: Ruft die neueste Version aus Ihrem Repository ab.
- **Änderungen in einer Datei vornehmen**: Verfolgt Ihre Änderungen als Entwurf, der zu einem Commit werden kann.
- **Änderungen speichern**: Erstellt einen Commit mit Ihren Änderungen.
- **Einen Branch erstellen**: Erstellt einen neuen Branch in Ihrem Repository.
- **Auf Ihrem Deployment-Branch veröffentlichen**: Erstellt einen Commit und pusht direkt auf Ihren Deployment-Branch.
- **Auf anderen Branches veröffentlichen**: Erstellt einen Commit und öffnet einen Pull Request.

<div id="when-to-use-branches">
  ### Wann Branches verwenden
</div>

Branches ermöglichen es Ihnen, an Änderungen zu arbeiten, ohne den Inhalt Ihrer Live-Website zu beeinträchtigen. Wenn Ihre Änderungen fertig sind, können Sie sie per Pull Request in Ihren Deployment-Branch mergen.

**Best Practice: Arbeiten Sie immer mit Branches.** So bleibt Ihre Live-Dokumentation stabil und ermöglicht saubere Review-Workflows.

<div id="creating-a-branch">
  ## Einen Branch erstellen
</div>

1. Wählen Sie den Branchnamen in der Editor-Toolbar aus (standardmäßig meist `main`).
2. Wählen Sie **New Branch**.
3. Geben Sie einen aussagekräftigen Namen für Ihren Branch ein, z. B. `update-getting-started` oder `fix-installation-steps`.
4. Wählen Sie **Create Branch**.

<Note>
  Möglicherweise müssen Sie Ihren neuen Branch im Dropdown-Menü auswählen, wenn der Editor nicht automatisch darauf umschaltet.
</Note>

<div id="saving-changes-on-a-branch">
  ### Änderungen in einem Branch speichern
</div>

Um Ihre Änderungen in einem Branch zu speichern, wählen Sie die Schaltfläche **Save Changes** in der oberen rechten Ecke des Editors. Dadurch wird ein Commit mit Ihren Änderungen erstellt und in Ihren Branch gepusht.

<div id="switching-branches">
  ### Branches wechseln
</div>

1. Wählen Sie den aktuellen Branchnamen in der Editor-Toolbar aus.
2. Wählen Sie im Dropdown-Menü den Branch aus, zu dem Sie wechseln möchten.

<Tip>
  Alle ungespeicherten Änderungen gehen beim Wechseln des Branches verloren. Stellen Sie sicher, dass Sie Ihre Arbeit speichern oder veröffentlichen, bevor Sie wechseln.
</Tip>

<div id="making-changes">
  ## Änderungen vornehmen
</div>

Bearbeiten Sie bestehende Inhalte, erstellen Sie neue Seiten und organisieren Sie die Struktur Ihrer Site im Web-Editor.

<div id="navigating-files">
  ### Dateien durchsuchen
</div>

Verwenden Sie den Dateiexplorer in der Seitenleiste, um Ihre Dokumentation zu durchsuchen, oder drücken Sie <kbd>Command</kbd> + <kbd>P</kbd> (<kbd>Ctrl</kbd> + <kbd>P</kbd> unter Windows), um nach Dateien zu suchen.

<div id="editing-content">
  ### Inhalte bearbeiten
</div>

Nehmen Sie Änderungen an Ihren Seiten im visuellen Modus oder im Markdown-Modus vor.

Drücken Sie im visuellen Modus <kbd>/</kbd>, um das Dropdown-Komponentenmenü zu öffnen. Fügen Sie Inhaltsblöcke, Hinweise, Codeblöcke und andere Komponenten hinzu, um Ihre Dokumentation anzupassen.

<div id="updating-navigation">
  ### Navigation aktualisieren
</div>

Fügen Sie Seiten in Ihrer Navigation hinzu, entfernen Sie sie oder ordnen Sie sie neu, indem Sie Ihre Datei `docs.json` bearbeiten:

1. Navigieren Sie im Dateiexplorer zu Ihrer `docs.json`.
2. Aktualisieren Sie die Eigenschaft `navigation`, um die gewünschte Navigationsstruktur zu erhalten. Weitere Informationen finden Sie unter [Navigation](/de/navigation).

Dieses Beispiel zeigt, wie Sie eine Themes-Seite zur Gruppe „Profile“ hinzufügen.

```json title="Adding a Themes page" highlight="18"
{
    "navigation": {
        "groups": [
            {
                "group": "Getting started",
                "pages": [
                    "index",
                    "quickstart",
                    "installation"
                ]
            },
            {
                "group": "Profile",
                "pages": [
                    "settings",
                    "account-types",
                    "dashboard",
                    "themes"
                ]
            }
        ]
    }
}
```

<div id="publishing-changes">
  ## Änderungen veröffentlichen
</div>

Wählen Sie die Schaltfläche **Publish**, um Ihre Änderungen zu speichern und zu veröffentlichen.

Wie Ihre Änderungen veröffentlicht werden, hängt von dem Branch ab, an dem Sie arbeiten:

- **Deployment-Branch**: Aktualisiert Ihre Live‑Site sofort.
- **Andere Branches**: Erstellt eine Pull-Request, damit Sie die Änderungen prüfen können, bevor sie live gehen.

<div id="pull-requests-and-reviewing-changes">
  ## Pull Requests und Änderungen prüfen
</div>

Pull Requests ermöglichen es dir, Änderungen aus deinem Branch vorzuschlagen, damit andere sie begutachten können, bevor sie in deine Live-Dokumentation übernommen werden. Das hilft, die Korrektheit sicherzustellen, und gibt deinem Team die Möglichkeit, an Inhalten zusammenzuarbeiten.

<Tip>
  Auch wenn du alleine arbeitest, sind Pull Requests hilfreich, um Änderungen vor der Veröffentlichung zu prüfen und eine nachvollziehbare Historie von Aktualisierungen zu behalten.
</Tip>

<div id="creating-a-pull-request-on-a-branch">
  ### Einen Pull Request aus einem Branch erstellen
</div>

Wenn du bereit bist, Änderungen aus deinem Branch zu veröffentlichen:

1. Stelle sicher, dass alle Änderungen in deinem Branch gespeichert sind.
2. Wähle **Publish Pull Request** oben rechts im Editor.
3. Füge einen Titel und eine Beschreibung für deinen Pull Request hinzu. Eine klare Benennung und Beschreibung hilft Reviewer:innen, deine Änderungen nachzuvollziehen.
4. Wähle **Publish Pull Request**.

<Frame>
  <img src="/images/editor/pull-request-light.png" alt="Hervorgehobene Schaltfläche „Publish Pull Request“ im Mintlify Web-Editor" className="block dark:hidden" />

  <img src="/images/editor/pull-request-dark.png" alt="Hervorgehobene Schaltfläche „Publish Pull Request“ im Mintlify Web-Editor" className="hidden dark:block" />
</Frame>

Der Web-Editor erstellt einen Pull Request in deinem Git-Repository und stellt einen Link zur Ansicht bereit.

<div id="reviewing-pull-requests">
  ### Pull Requests prüfen
</div>

Sobald dein Pull Request erstellt wurde:

1. **Änderungen prüfen**: Du und dein Team könnt den Pull Request bei eurem Git-Anbieter wie GitHub oder GitLab überprüfen.
2. **Weitere Änderungen vornehmen**: Nimm nach der Überprüfung etwaige Anpassungen im Web-Editor vor. Durch Speichern werden sie im Pull Request aktualisiert.
3. **Mergen, wenn bereit**: Wenn dein Pull Request bereit ist, merge ihn, um die Änderungen auf deiner Live-Dokumentationsseite zu deployen.

<div id="troubleshooting">
  ## Fehlerbehebung
</div>

Hier finden Sie Lösungen für häufige Probleme, die im Web-Editor auftreten können.

<AccordionGroup>
  <Accordion title="Änderungen werden nach dem Veröffentlichen nicht angezeigt">
    **Mögliche Ursachen:**

    - Die Bereitstellung ist noch nicht abgeschlossen
    - Caching-Probleme in Ihrem Browser

    **Lösungen:**

    1. Prüfen Sie den Bereitstellungsstatus in Ihrem Mintlify-Dashboard.
    2. Führen Sie ein Hard-Reload in Ihrem Browser durch (<kbd>Ctrl</kbd> + <kbd>F5</kbd> oder <kbd>Cmd</kbd> + <kbd>Shift</kbd> + <kbd>R</kbd>).
    3. Leeren Sie den Browser-Cache.
  </Accordion>

  <Accordion title="Berechtigungsfehler beim Veröffentlichen">
    **Mögliche Ursachen:**

    - Unzureichende Berechtigungen für das Git-Repository
    - Authentifizierungsprobleme mit Ihrem Git-Anbieter

    **Lösungen:**

    1. Vergewissern Sie sich, dass Sie die richtigen Zugriffsrechte für das Repository haben.
    2. Prüfen Sie, ob Ihre Git-Integration korrekt konfiguriert ist.
    3. Lesen Sie die Dokumentation zu [Editor Permissions](/de/advanced/dashboard/permissions).
  </Accordion>

  <Accordion title="Probleme beim Laden des Editors">
    **Mögliche Ursachen:**

    - Netzwerkverbindungsprobleme
    - Große Dokumentations-Repositories

    **Lösungen:**

    1. Überprüfen Sie Ihre Internetverbindung.
    2. Laden Sie die Seite neu und versuchen Sie es erneut.
    3. Kontaktieren Sie den Support, wenn das Problem weiterhin besteht.
  </Accordion>
</AccordionGroup>
